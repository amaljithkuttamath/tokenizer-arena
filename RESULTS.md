# Tokenizer Arena: Benchmark Results

These results were generated by running `tokenizer-arena` against four categories of input text. Each table compares how cl100k_base (GPT-4/Claude), o200k_base (GPT-4o), p50k_base (GPT-3/Codex), and r50k_base (GPT-3 legacy) handle the same input.

## English Prose

A paragraph about transformer models. 701 bytes, 100 words.

| Model | Encoding | Tokens | Bytes/Token | Tokens/Word |
|-------|----------|--------|-------------|-------------|
| GPT-4 / Claude | cl100k_base | 127 | 5.52 | 1.27 |
| GPT-4o | o200k_base | 126 | 5.56 | 1.26 |
| GPT-3 / Codex | p50k_base | 133 | 5.27 | 1.33 |
| GPT-3 (legacy) | r50k_base | 133 | 5.27 | 1.33 |

**Takeaway:** For standard English text, all four encodings perform similarly. The newer vocabularies (cl100k, o200k) are slightly more efficient, needing about 5% fewer tokens than the older ones. Most English words map to a single token, with tokens/word ratios close to 1.3 across the board.

## Python Code

A multi-head attention implementation. 1513 bytes, 141 words.

| Model | Encoding | Tokens | Bytes/Token | Tokens/Word |
|-------|----------|--------|-------------|-------------|
| GPT-4 / Claude | cl100k_base | 410 | 3.69 | 2.91 |
| GPT-4o | o200k_base | 413 | 3.66 | 2.93 |
| GPT-3 / Codex | p50k_base | 576 | 2.63 | 4.09 |
| GPT-3 (legacy) | r50k_base | 738 | 2.05 | 5.23 |

**Takeaway:** Code is where the generational gap becomes obvious. cl100k and o200k need roughly 410 tokens while r50k needs 738, nearly 1.8x more. The newer tokenizers were trained on code-heavy corpora and learned to merge common patterns like `self.`, `def `, and indentation into single tokens. If you are working with code-heavy prompts, model choice matters for cost.

## JSON Payload

A sample API request with nested objects. 613 bytes, 64 words.

| Model | Encoding | Tokens | Bytes/Token | Tokens/Word |
|-------|----------|--------|-------------|-------------|
| GPT-4 / Claude | cl100k_base | 189 | 3.24 | 2.95 |
| GPT-4o | o200k_base | 189 | 3.24 | 2.95 |
| GPT-3 / Codex | p50k_base | 230 | 2.67 | 3.59 |
| GPT-3 (legacy) | r50k_base | 264 | 2.32 | 4.12 |

**Takeaway:** JSON sits between prose and code. The structural characters (braces, colons, quotes) and repeated key patterns benefit from the larger vocabularies. cl100k and o200k are identical here, both about 28% more efficient than r50k. If your application sends structured payloads to the API, newer tokenizers save meaningful token budget.

## Multilingual Text

English, Chinese, Japanese, Arabic, and emoji. 302 bytes, 27 words.

| Model | Encoding | Tokens | Bytes/Token | Tokens/Word |
|-------|----------|--------|-------------|-------------|
| GPT-4 / Claude | cl100k_base | 103 | 2.93 | 3.81 |
| GPT-4o | o200k_base | 69 | 4.38 | 2.56 |
| GPT-3 / Codex | p50k_base | 139 | 2.17 | 5.15 |
| GPT-3 (legacy) | r50k_base | 139 | 2.17 | 5.15 |

**Takeaway:** This is where the differences are most dramatic. o200k_base (GPT-4o) uses only 69 tokens compared to 139 for the older encodings, a 2x improvement. The 200K vocabulary was specifically expanded to handle CJK characters, Arabic script, and emoji more efficiently. cl100k sits in the middle at 103 tokens. For multilingual workloads, o200k offers a substantial cost advantage.

## Summary

| Input Type | cl100k | o200k | p50k | r50k | Best |
|------------|--------|-------|------|------|------|
| English Prose | 127 | 126 | 133 | 133 | o200k |
| Python Code | 410 | 413 | 576 | 738 | cl100k |
| JSON | 189 | 189 | 230 | 264 | tie (cl100k/o200k) |
| Multilingual | 103 | 69 | 139 | 139 | o200k |

The pattern is clear: newer tokenizers are universally better, with the biggest gains on non-English text and code. o200k's doubled vocabulary pays off most for multilingual content, while cl100k and o200k are nearly identical for English and structured data.
